# @damatjs/mcp 0.1.0

> First release: a dependency-free Model Context Protocol server that lets an AI assistant discover Damat modules in a registry and install them into an app, by shelling out to the `damat` CLI.

## What changed

This is the first published version of `@damatjs/mcp`, so there is no previous
behavior to migrate from. Before it, installing a Damat module from an AI
assistant meant the assistant guessing at file copies, `damat.config.ts` edits,
env syncing, and `bun add` calls by hand — there was no audited, tool-shaped path
for it to follow.

`@damatjs/mcp` is an [MCP](https://modelcontextprotocol.io) **server** that
bridges two things that already exist in Damat — the **module registry** (an
index JSON described by `@damatjs/module`) and the **`damat module add`** command
(from `@damatjs/damat-cli`) — and exposes them to any MCP client (Claude Code,
Claude Desktop, Cursor, …). The assistant never touches the filesystem to install
a module; it calls a tool, the tool runs the audited CLI path, and the CLI does
the copy + config registration + env sync + package install.

It is intentionally **dependency-free**: it speaks the MCP stdio transport
(newline-delimited JSON-RPC 2.0) directly and runs straight from source under Bun
with no build step. The package ships the `damat-mcp` bin plus `src/` and a
sample `registry.example.json`.

**The five tools** (the order the assistant usually chains them):

| Tool | Inputs | What it does |
|------|--------|--------------|
| `list_modules` | — | Loads the configured registry index and returns a summary per module (ref, description, latest version, verification, owner). |
| `search_modules` | `query` | Same, filtered by a case-insensitive match against ref, description and keywords. |
| `module_info` | `ref` | Full registry details for one ref (e.g. `damatjs/user@0.2.0`) — read before installing. |
| `list_installed` | `dir?` (default `src/modules`) | Scans the app's modules directory for subdirectories with a `module.json` manifest. |
| `add_module` | `source`, `name?`, `dir?`, `force?` | Runs `damat module add <source>` in the app dir. `source` may be a registry ref, a local path, a github shorthand (`owner/repo`), or a git URL. |

The registry tools degrade gracefully: with no registry configured they return
guidance, and `add_module` still works for git/local-path sources.

**Configuration** is entirely via environment variables (set them in
`.mcp.json` → `env`):

| Variable | Default | Description |
|----------|---------|-------------|
| `DAMAT_MODULE_REGISTRY` | — | Registry index location: an http(s) URL to the index JSON, a path to a `registry.json`, or a directory containing one. Unset → registry-less mode. |
| `DAMAT_APP_DIR` | `process.cwd()` | Working directory of the target app; installs and "installed" scans run here. |
| `DAMAT_CLI` | `damat` | Command used to invoke the CLI. May include args, e.g. `bun /abs/packages/cli/damat/src/cli.ts`. |
| `DAMAT_MODULE_VERIFY` | `warn` | The install-time verification policy, read by the CLI/`@damatjs/module`: `off` \| `warn` \| `require`. |

**Transport & protocol.** stdio, one JSON-RPC message per `\n`-terminated UTF-8
line; malformed lines are ignored. On `initialize` the server echoes the
client's `protocolVersion` (falling back to `2025-06-18`) and advertises the
`tools` capability. It handles `initialize`, `ping`, `tools/list`, and
`tools/call` (plus `notifications/initialized` as a no-op); unknown request
methods return `-32601 Method not found`. Handler errors come back as
`isError: true` tool results, not JSON-RPC errors, so the assistant can recover.

## Added
- The `damat-mcp` bin and the `@damatjs/mcp` package — a dependency-free MCP
  server that runs under Bun with no build step.
- Five MCP tools: `list_modules`, `search_modules`, `module_info`,
  `list_installed`, `add_module`.
- Four configuration env vars: `DAMAT_MODULE_REGISTRY`, `DAMAT_APP_DIR`,
  `DAMAT_CLI`, `DAMAT_MODULE_VERIFY`.
- A sample `registry.example.json` used by the repo's `.mcp.json` and for local
  testing.

## Action required

The MCP server is opt-in; nothing changes for an app unless you wire it up. To
start using it, add an `.mcp.json` to your project root and point a server entry
at the `damat-mcp` bin:

```json
{
  "mcpServers": {
    "damat-modules": {
      "command": "bun",
      "args": ["run", "packages/mcp/bin/damat-mcp.ts"],
      "env": {
        "DAMAT_MODULE_REGISTRY": "https://registry.damatjs.com/index.json",
        "DAMAT_APP_DIR": ".",
        "DAMAT_CLI": "damat"
      }
    }
  }
}
```

In a published app, add the dependency and point `args` at the installed bin:

```bash
bun add -D @damatjs/mcp
```

```json
{ "command": "bunx", "args": ["damat-mcp"] }
```

Then ask the assistant to find and install a module; it will chain
`search_modules` → `module_info` → `add_module`, and tell you to run migrations
afterward.

Smoke-test it without a client by piping newline-delimited JSON-RPC at the bin:

```bash
printf '%s\n' \
  '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2025-06-18","capabilities":{}}}' \
  '{"jsonrpc":"2.0","id":2,"method":"tools/list"}' \
  '{"jsonrpc":"2.0","id":3,"method":"tools/call","params":{"name":"list_modules","arguments":{}}}' \
| DAMAT_MODULE_REGISTRY=packages/mcp/registry.example.json \
  bun run packages/mcp/bin/damat-mcp.ts
```
