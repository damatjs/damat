# @damatjs/orm-pg 0.1.0

> First published minor release: the entity-manager layer is hardened — an optional `models` config, registry-derived transaction accessors, a transaction isolation-level allow-list, a rollback-error guard, and typed (unwrapped) repository reads.

## What changed

`0.1.0` is the stabilized minor release. The execution stack from the pre-alpha
line is unchanged in shape; the changes are concentrated in the manager,
transaction, repository, and config layers.

### 1. `PgEntityManagerConfig` gains an optional `models` map

Before, the config was `{ pool, logger? }` and you had to register every model
after construction. `0.1.0` adds an optional `models` map (and makes the config
generic over it), so models can be registered up front.

```ts
// Before — register after construction only
const em = new PgEntityManager({ pool });
em.registerModel("user", User);
em.registerModel("post", Post);

// After — register up front (registerModel still works too)
const em = new PgEntityManager({
  pool,
  models: { user: User, post: Post },
});
```

```ts
// types/config.ts
export interface PgEntityManagerConfig<
  TModels extends Record<string, ModelDefinition> = Record<string, ModelDefinition>,
> {
  pool: Pool;
  logger?: ILogger;
  models?: TModels; // ← added
}
```

The constructor iterates `config.models` and calls `registerModel(name, model)`
for each before initializing repositories.

### 2. `em.<model>` accessors, and `tx.<model>` accessors derived from the registry

The entity manager now installs a `_defineModelAccessor(name)` getter during
`registerModel`, so `em.user` returns the cached repository for `"user"` (the
same instance as `em.repo("user")`) — mirroring the `tx.<model>` accessors. The
getter is skipped when the model name would shadow a real method/field.

The transactional manager previously **required** a `modelsConfig` object and
derived its `tx.<model>` accessors from that object's keys. Now `modelsConfig` is
optional; when omitted (which is how `PgEntityManager` calls it), accessors are
derived from `modelRegistry.getModelNames()`.

```ts
// Before — transaction passed the manager's per-instance models config
new TransactionalEntityManager(this.modelRegistry, ctx, this.logger, this.modelsConfig);

// After — accessors come from the registry; no per-instance config needed
new TransactionalEntityManager(this.modelRegistry, ctx, this.logger);
```

### 3. Transaction isolation level is validated against an allow-list

The isolation level is interpolated into the `SET TRANSACTION ISOLATION LEVEL …`
statement, so `0.1.0` adds a runtime guard rejecting any value not in the
PostgreSQL set, defending against untyped config or JSON input that bypasses the
type system.

```ts
const VALID_ISOLATION_LEVELS = new Set([
  "READ UNCOMMITTED", "READ COMMITTED", "REPEATABLE READ", "SERIALIZABLE",
]);
// throws `Invalid transaction isolation level: "<value>"` otherwise
```

### 4. A failed ROLLBACK no longer masks the original error

`TransactionManager.run` previously called `await ctx.rollback()` bare on the
error path; if the rollback itself threw, that secondary error replaced the real
cause. Now the rollback is wrapped — a rollback failure is logged and the
original error is re-thrown.

### 5. Repository reads return unwrapped, typed values

`PgRepository.findMany` and `findManyByIds` previously returned the raw client
result typed as `Promise<any>`. They now return `Promise<T[]>` by unwrapping
`.rows`, consistent with the other read methods.

```ts
// Before
async findMany(opt = {}): Promise<any> { return this.client.findMany(opt); }

// After
async findMany(opt = {}): Promise<T[]> { return (await this.client.findMany(opt)).rows; }
```

## Added
- Optional `models` map on `PgEntityManagerConfig` (config is now generic over it).
- `em.<model>` dynamic repository accessors on `PgEntityManager`.
- A transaction isolation-level allow-list with a descriptive error.

## Changed / improved
- `TransactionalEntityManager`'s `modelsConfig` argument is now optional; accessors
  fall back to the model registry's names.
- `TransactionManager.run` guards a failing `ctx.rollback()` so the original error
  is preserved and the rollback failure is logged.
- `PgRepository.findMany` / `findManyByIds` return `Promise<T[]>` (unwrapped rows)
  instead of `Promise<any>` (the raw client result).

## Breaking
- None for typical consumers. `findMany` / `findManyByIds` now resolve to the
  rows array rather than the raw `{ rows, rowCount, descriptor }` client result;
  code that read `.rows` off the repository (rather than the client) would need
  adjustment, but the repository methods were already documented to return rows.

## Action required

None — drop-in upgrade.

- Registering via `models` is **opt-in**; `registerModel(name, model)` continues
  to work exactly as before.
- If you ever pass `isolationLevel` from untyped/JSON config, ensure it is one of
  `READ UNCOMMITTED` / `READ COMMITTED` / `REPEATABLE READ` / `SERIALIZABLE`;
  other values now throw at `begin` time.
