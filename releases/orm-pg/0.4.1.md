# @damatjs/orm-pg 0.4.1

> Repository lookups now fall back to the table name — cascade deletes work on
> snake_case tables whose registry key (the camelCase accessor) doesn't match the
> name the cascade walker carries.

## What changed

`PgEntityManager.getRepository(modelName)` and
`TransactionalEntityManager.getRepository(modelName)` resolved models only by
their registry key — the camelCase accessor derived from the table name. Cascade
delete traverses relations using **table names** (e.g. `member_accounts`), so on
any table whose name isn't already its accessor key the lookup missed and the
cascade failed.

- **Before:** `modelRegistry.get(modelName)` — accessor-key lookup only.
- **After:** `modelRegistry.get(modelName) ?? modelRegistry.getByTableName(modelName)` —
  the table name is tried when the key misses.

A regression test covers cascade delete through a snake_case child table
(`cascade.integration.test.ts`, gated on `DATABASE_URL`).

## Upgrade notes

Nothing to do — lookups that worked before still resolve the same model; lookups
that previously threw for snake_case tables now succeed.
