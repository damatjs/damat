# @damatjs/codegen 0.1.4

> Codegen graduates from a types-only generator to the engine behind the
> **codegen-first** module workflow: it still emits row types + zod, and now also
> writes the typed `registry.ts` and scaffolds-once a full per-table CRUD slice
> (steps, workflows, and split `api/routes`).

## What changed

`@damatjs/codegen` previously turned a `ModuleSchema` into TypeScript row types
and zod schemas and nothing else. It now drives the codegen-first authoring loop:
after writing the type files it augments the typed module `registry.ts` and, for
every table in the schema, scaffolds the route → workflow → step CRUD slice the
framework expects — but only the first time.

`generateCrudScaffold` loops the schema's tables and writes, per table:

- **steps** — `create` / `update` / `delete` / `find` / `findMany` + an index,
- **workflows** — the matching set + an index,
- **api/routes** — the collection route (`api.ts` / `validator.ts` / `query.ts` /
  `middleware.ts` / `route.ts`) and the `[id]` route.

Every write goes through `writeOnce`: if the target file already exists it is
**skipped**, so re-running codegen after you have edited a generated step or route
never clobbers your work. Only the row types under `types/` are regenerated
wholesale.

## Added

- `generateCrudScaffold(...)` and the per-table step / workflow / route templates.
- `registryAugmentation` — writes the typed `registry.ts` so `getModule("<name>")`
  is typed for the module's models.
- Orchestration in `runModuleCodegen` chaining: row types + zod → `registry.ts` →
  CRUD scaffold.

## Changed / improved

- The package is no longer types-only. Both `damat codegen` (app) and
  `damat module codegen` (standalone module) produce the full slice through it.

## Breaking

- None. The scaffold is additive and `writeOnce`-guarded; the type/zod output is
  unchanged. (Published earlier as `@damatjs/orm-codegen`; the import path is now
  `@damatjs/codegen`.)

## Action required

- None — drop-in. Re-run `damat codegen` / `damat module codegen` to materialize
  the per-table steps, workflows, and routes, then edit them in place (they
  survive re-runs). Keep treating `types/` as generated output.
