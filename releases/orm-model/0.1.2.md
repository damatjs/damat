# @damatjs/orm-model 0.1.2

> Relation targets can now be a plain table-name string, with foreign keys inferred by convention — no model import, no circular-import dance, and cross-module references become possible.

## What changed

Before `0.1.2`, the target of a relation (`belongsTo` / `hasOne` / `hasMany`) had
to be a **live model**: either a direct import of the target `ModelDefinition` or
a lazy thunk that returns it.

```ts
// Before — model import or lazy thunk
import { User } from "./user";

const Order = model("order", {
  user: columns.belongsTo(User).onDelete("CASCADE"),
});

// or, to dodge circular imports between two files:
const Order = model("order", {
  user: columns.belongsTo(() => User).onDelete("CASCADE"),
});
```

`0.1.2` adds a third target form: a **table-name string**. The foreign key
column, FK constraint, and relation metadata are inferred by convention
(`<targetTable>_id` referencing `id`), so you no longer import the target model
at all.

```ts
// After — reference the target table by name
const Order = model("order", {
  user: columns.belongsTo("user").onDelete("CASCADE"),
});

const User = model("user", {
  orders: columns.hasMany("order").mappedBy("user"),
});
```

The type that backs this is `ModelTarget = ModelDefinition | LazyModel | string`.
String targets are resolved through the global table-name → model registry that
every `ModelDefinition` self-registers into on construction.

What this unlocks:

- **No imports of the target model** — schema generation (FK column, constraint,
  relation schema, codegen) uses the string name directly via
  `getModuleTargetTable()` without resolving the live model, so import order does
  not matter for building schemas.
- **No circular imports** — two tables that reference each other can each name the
  other as a string instead of importing it.
- **Cross-module relations** — a relation can reference a table owned by another
  module without importing that module's model.

Resolution to the live model is only needed at query time (in `@damatjs/orm-pg`'s
relation resolver); at that point the named model must already be loaded, or
`resolveModuleTarget` throws a descriptive "not found in registry" error.

## Added
- String table-name relation targets for `belongsTo`, `hasOne`, and `hasMany`
  (`belongsTo("users")`), with FK columns/constraints inferred by convention.
- `ModelTarget` now includes `string`; resolution is backed by the global model
  registry (`registerModel` / `getRegisteredModel` / `hasRegisteredModel`).

## Changed / improved
- `getModuleTargetTable()` returns a string target verbatim, so FK/constraint/
  relation-schema generation no longer resolves the live model — enabling
  cross-module and circular references at schema-build time.

## Breaking
- None. This is purely additive. The existing model-reference form
  (`belongsTo(User)`) and the lazy-thunk form (`belongsTo(() => User)`) both still
  compile and behave exactly as before — `ModelTarget` is a union that keeps them.

## Action required

None — drop-in upgrade. The new string form is **opt-in**.

To adopt string targets in existing relations (optional):

1. Replace the model import / lazy thunk with the target's table name:
   ```ts
   - user: columns.belongsTo(User)
   - user: columns.belongsTo(() => User)
   + user: columns.belongsTo("user")
   ```
   Do the same for `hasOne` / `hasMany` (e.g. `hasMany("order").mappedBy("user")`).
2. Remove the now-unused model imports for those targets.
3. The inferred FK column is `<targetTable>_id` referencing `id`. If your existing
   relation used a different FK or referenced column, keep that explicit via
   `.link({ foreignKey, reference })` so the generated schema is unchanged.
4. Ensure every model whose table you reference by string is imported/loaded
   before any query traverses the relation (schema generation does not need it;
   query-time resolution does).
