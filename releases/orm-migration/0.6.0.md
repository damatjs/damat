# @damatjs/orm-migration 0.6.0

> Concurrent deploys serialize on a Postgres advisory lock, statements that
> can't run in a transaction (`CREATE INDEX CONCURRENTLY`, `ALTER TYPE … ADD
> VALUE`, …) execute outside one, and the status API is keyed by module name.

## What changed

### Advisory-lock serialization

`runMigrations` acquires `pg_advisory_lock()` before bootstrap or any migration
and releases it in a `finally`. Two deploys landing at once no longer race on
tracker-table creation or interleave migrations. Lock acquisition failures are
caught and returned as structured `ModuleMigrationResult`s — the "always returns
results, never raw rejects" contract holds.

### Non-transactional statements

`executeMigration` detects `CREATE INDEX CONCURRENTLY`, `DROP INDEX
CONCURRENTLY`, `REINDEX … CONCURRENTLY`, and `ALTER TYPE … ADD VALUE` and runs
those migrations **without** a `BEGIN`/`COMMIT` wrapper (they are illegal inside
one). Errors from these skip `ROLLBACK` — there is no open transaction — so a
failed rollback can no longer mask the original migration error.

### Status keyed by module name

The migration-status functions take an `OrmModuleContainer` (name-keyed module
map) instead of raw path arrays; tracker queries key on the module `name`,
discovery on its `resolve` path.

## Upgrade notes

- Callers of the status API pass the module container now (the `damat` CLI and
  `@damatjs/orm-cli` 0.6.0 already do).
- Migrations containing `CONCURRENTLY`/`ADD VALUE` statements that previously
  failed inside transactions now run — review any workarounds you added for
  that.
