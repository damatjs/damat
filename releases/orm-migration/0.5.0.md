# @damatjs/orm-migration 0.5.0

> Per-module migration discovery now sorts numerically by timestamp (with the
> filename as tiebreaker), matching all-module discovery — the same migrations
> run in the same order no matter which entry point discovered them.

## What changed

`discoverModuleMigrations` sorted migration files **lexically** by filename,
while `discoverAllMigrations` sorted **numerically** by the extracted timestamp.
A migration without a timestamp in its name (extracted as `0`) could therefore
be ordered differently depending on which discovery path ran — inconsistent
ordering across per-module and cross-module runs.

- **Before:** `.sort()` — lexical.
- **After:** `.sort((a, b) => a.timestamp - b.timestamp || a.name.localeCompare(b.name))` —
  oldest timestamp first, stable filename tiebreaker, identical to
  `discoverAllMigrations`.

## Upgrade notes

Nothing to do for migrations with standard timestamped names — their order is
unchanged. If you carried a timestamp-less migration file, verify it still sorts
where you expect (it now sorts first, before all timestamped files).
