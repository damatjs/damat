# @damatjs/orm-migration 0.1.0

> First published minor release: the module-based PostgreSQL migration runtime for
> the Damat ORM — discover, generate, run, and track migrations per module —
> stabilized for general use.

## What changed

`0.1.0` is the baseline minor release of `@damatjs/orm-migration`. It sits between
the pure schema engine ([`@damatjs/orm-processor`](../../packages/orm/processor/README.md))
and a live database: each application module owns its own `migrations/` folder of
timestamped `.sql` files, and this package finds those files, generates new ones
from a module's models, applies the pending ones transactionally against a `pg`
`Pool` you supply, and records what has been applied in a single shared
`_damat_migration_logs` table.

At this baseline the package provides four cooperating layers:

- **Discovery** — `discoverModuleMigrations(resolver)` scans `<resolver>/migrations/`
  for `Migration*.sql` files and returns sorted `MigrationInfo[]`;
  `discoverAllMigrations(resolvers[])` aggregates across modules in chronological
  order; `discoverModels(resolver)` dynamically `import()`s a module and returns
  its `ModelDefinition[]` (throws when a module exposes no `models`).
- **Generation (offline, never touches the DB)** — `createInitialMigration` emits a
  full baseline `CREATE` SQL file from all of a module's models and writes the first
  `schema-snapshot.json`; `createDiffMigration` diffs the current models against the
  saved snapshot and writes only the changes (returning `filePath: null` when there
  are none, unless `force`). `createMigration` is a convenience router that picks
  initial vs diff based on whether a snapshot already exists.
- **Execution** — `runMigrations(pool, container)` ensures the tracker table exists,
  runs idempotent `bootstrapDatabase` prerequisites (the `pgcrypto` extension plus a
  `generate_id(prefix)` helper), then for each module computes the pending set and
  applies each `.sql` file in its own `BEGIN`/`COMMIT` transaction, stopping that
  module on first failure. Returns one `ModuleMigrationResult` per module.
- **Tracking** — `MigrationTracker` owns `_damat_migration_logs`, keying applied
  migrations by `(module, name)` so two modules can each have a migration named
  `..._Initial` without collision. Status helpers (`getMigrationStatus`,
  `getModuleMigrationStatus`) report applied vs pending counts per module.

Design invariants set at this baseline: **you bring the pool** (the package never
opens or closes connections), **each migration runs in its own transaction**,
**generation is offline** (it diffs models against the on-disk snapshot, never the
DB), and **the runner only applies forward** — `recordReverted` exists for a
caller-implemented revert flow but `runMigrations` never calls it.

## Added
- The full discovery / generation / execution / tracking surface described above,
  exported from the package root: `createMigration`, `createInitialMigration`,
  `createDiffMigration`, `discoverModuleMigrations`, `discoverAllMigrations`,
  `discoverModels`, `runMigrations`, `getMigrationStatus`,
  `getModuleMigrationStatus`, `bootstrapDatabase`, the `MigrationTracker` class, and
  the `log` / `separator` / `successBanner` / `errorBanner` logging helpers.

## Changed / improved
- Stabilized core functionality, expanded test coverage, and resolved known issues
  relative to the `0.0.x` pre-alpha line.

## Breaking
- None — first stable minor.

## Action required

None — drop-in for new projects. Add the dependency and drive it from the ORM CLI
or directly:

```bash
bun add @damatjs/orm-migration
```
