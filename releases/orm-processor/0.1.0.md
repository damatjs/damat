# @damatjs/orm-processor 0.1.0

> First published minor release: the pure schema engine (snapshot, diff, SQL generation) stabilized for general use — and a correctness fix to how a *changed* index or foreign key is re-created.

## What changed

`0.1.0` is the baseline minor release of `@damatjs/orm-processor`. It is the
low-level, database-free engine that turns serialized schema definitions into
migrations. It does three things, all as pure functions over the `ModuleSchema`
shape from `@damatjs/orm-type`:

- **Snapshot** (`snapshot/index.ts`) — `loadSnapshot` / `saveSnapshot` /
  `snapshotExist` persist a module's schema as `schema-snapshot.json` in its
  migrations directory. This is the only part of the package that touches disk; a
  missing file is read as an empty baseline so the first diff produces a full
  "create everything" change set.
- **Diff** (`diff/`) — `diffSchemas(previous, current)` compares two schemas and
  returns a priority-sorted `SchemaDiff` of `SchemaChange` records (tables,
  columns, indexes, foreign keys, native enums), plus non-fatal warnings for
  destructive operations. `reverseDiff` inverts a forward diff for `down`
  migrations (drops are intentionally not reconstructed).
- **SQL generation** (`sqlGenerator/`) — `generateFromDiff(diff)` emits ordered
  PostgreSQL `up` DDL for an incremental migration; `generateFromSnapshot(schema)`
  emits a full baseline migration from a single schema. `safeMode` (default
  `true`) adds `IF EXISTS` / `IF NOT EXISTS` guards; identifiers are always
  double-quoted and schema-qualified.

The single numeric priority on each change is the contract between the diff and
SQL layers: `diffSchemas` sorts once, and the generator emits statements in that
order without re-reasoning about dependencies (enums before tables, tables before
FKs, drops last and in reverse dependency order).

### Re-add ordering fix for changed indexes / foreign keys

Because PostgreSQL cannot alter an index or a foreign-key constraint in place, a
**changed** index or FK is expressed as a drop followed by a re-add. Earlier in
the pre-release line, the re-add carried the *same* priority as a brand-new
add — `ADD_INDEX` (40) / `ADD_FOREIGN_KEY` (50) — which are far **lower** than the
matching drop priorities (`DROP_INDEX` 110 / `DROP_FOREIGN_KEY` 100). After the
single priority sort, the re-create therefore sorted *before* the drop, so the
generated `up` SQL tried to create the new index/FK while the old one still
existed.

`0.1.0` adds two dedicated priorities so the re-add lands just after its drop:

```ts
// src/diff/priority.ts
DROP_FOREIGN_KEY:  100,
READD_FOREIGN_KEY: 105,   // ← new: re-add a changed FK after the drop
DROP_INDEX:        110,
READD_INDEX:       115,   // ← new: re-add a changed index after the drop
```

```ts
// before — diff/indexes.ts (changed index)
priority: PRIORITY.ADD_INDEX        // 40  → sorted before the drop (broken)
// after
priority: PRIORITY.READD_INDEX      // 115 → sorted after the drop

// before — diff/foreignKeys.ts (changed FK)
priority: PRIORITY.ADD_FOREIGN_KEY  // 50  → sorted before the drop (broken)
// after
priority: PRIORITY.READD_FOREIGN_KEY // 105 → sorted after the drop
```

A purely *added* index or FK still uses the normal `ADD_INDEX` / `ADD_FOREIGN_KEY`
priority; only the re-add half of a change moved.

## Added

- The full schema engine described above: `loadSnapshot` / `saveSnapshot` /
  `snapshotExist`; `diffSchemas`, `reverseDiff`, and the per-concern diff
  namespaces (`tablesDiff`, `columnsDiff`, `indexesDiff`, `foreignKeysDiff`,
  `enumsDiff`, `utilsDiff`, `priorityDiff`); `generateMigration.generateFromDiff`
  / `generateFromSnapshot`, `changeSqlGenerator.generateChangeSQL` /
  `generateDescription`, and the per-concern SQL emitter namespaces.
- The `SchemaDiff` / `SchemaChange` types and the
  `MigrationGeneratorOptions` / `GeneratedMigration` /
  `CreateDiffMigrationOptions` / `DiffMigrationResult` shapes.
- `READD_FOREIGN_KEY` (105) and `READD_INDEX` (115) priority constants.

## Changed / improved

- A **changed** index or foreign key now re-creates *after* its drop, fixing
  invalid `up` SQL that previously emitted the re-create before removing the old
  definition (see above).

## Breaking

- None. The public API surface (function names, signatures, exported types) is
  unchanged from the pre-release line; the priority fix only corrects emitted
  statement ordering.

## Action required

None — drop-in upgrade. New consumers add the dependency and import from the
package root:

```bash
bun add @damatjs/orm-processor
```

If you keep schema snapshots from a pre-release version, no migration of the
snapshot file is needed — the format is unchanged. Any migration you previously
generated for a *changed* index/FK should be re-generated to pick up the correct
drop-then-recreate ordering.
