# @damatjs/link 0.1.3

> First release: cross-module links — declare a relationship between two models in different modules, get an auto-generated junction table that migrates and type-generates through the existing pipelines, and create/dismiss/query the relationship at runtime.

## What changed

This is the first published version of `@damatjs/link`, so there is no previous
behavior to migrate from — before it, Damat modules were fully isolated vertical
slices with no first-class way to relate to one another. Modules don't share
tables and can't foreign-key into each other, so "a user belongs to
organizations" or "a product has categories" had no supported home.

`@damatjs/link` introduces the **link**: a relationship declared _outside_ both
modules (in the app's `src/links/`) that generates a junction table to store the
relationship and exposes a service to manage and traverse it. Neither module
imports the other; the relationship is first-class and decoupled.

A link is authored with `defineLink`:

```ts
// src/links/user/models/user-organization.ts
import { defineLink } from "@damatjs/framework";

export default defineLink(
  { module: "user", model: "user", field: "users" },
  { module: "organization", model: "organization", field: "organizations" },
);
```

Here `module` is the module id (the `getModule` key) and `model` is the **key in
that module's `models` map** (the service accessor — `user`, not the table
`users`). `field` is the name the linked side is exposed as; `isList` (default
`true`) marks the side as a collection; `primaryKey` (default `"id"`) is the
column whose value the junction stores.

This generates a `user_organization` junction table with `id`, `user_id`,
`organization_id`, `created_at` / `updated_at` / `deleted_at`, a **unique** index
on `(user_id, organization_id)`, and a per-foreign-key-column index. No real
cross-module database foreign keys are emitted by default (module isolation);
opt in with `{ database: { foreignKeys: true } }`. Extra junction columns can be
added with `{ database: { extraColumns: { … } } }`, and the table name overridden
with `{ pivotTable: "…" }` / id prefix with `{ idPrefix: "…" }`.

Links live under an owner-module folder that mirrors a module's shape
(`models/`, `index.ts`, `migrations/`):

```ts
// src/links/user/index.ts
import { collectLinkModels } from "@damatjs/framework";
import userOrganization from "./models/user-organization";

export const links = [userOrganization];
export const models = collectLinkModels(links);
```

```ts
// src/links/index.ts
import { defineLinkModule } from "@damatjs/framework";
import { links as userLinks } from "./user";

export const links = [...userLinks];
export default defineLinkModule(links);
```

The feature is wired into the rest of Damat with **no special-casing**: a
junction table is an ordinary ORM `ModelDefinition` and the link module is an
ordinary `defineModule` instance, so migrations, snapshots, and codegen flow
through the existing pipelines.

- **Config.** A new `links` field in `damat.config.ts` points at the links root:

  ```ts
  export default defineConfig({
    projectConfig: {/* … */},
    modules: {
      user: { resolve: "./src/modules/user" },
      organization: { resolve: "./src/modules/organization" },
    },
    links: "./src/links",
  });
  ```

- **Migrations.** Each owner directory is its own migration module, registered as
  `link:<owner>`. `damat-orm migrate:create link:user` generates the junction
  migration into `links/user/migrations`, and `migrate:up` creates the table.

- **Types.** Links emit no junction types. Instead `generate:types <module>`
  weaves a sibling `<table>.links.ts` into each **linked** module, merging the
  linked entity onto the base interface via declaration merging (the
  model-generated `<table>.ts` stays untouched):

  ```ts
  // modules/user/types/users.links.ts  (auto-generated)
  import type { Organizations } from "../../organization/types";
  declare module "./users" {
    interface Users {
      organizations?: Organizations[];
    }
  }
  ```

  So `Users` carries `organizations`, `Organizations` carries `users`.

- **Runtime.** The framework boot registers the aggregated links dir as the
  single `link` module and injects `getModule` into the link service for
  cross-module hydration. `getModule("link")` returns a service with:

  ```ts
  const link = getModule("link");

  // create is idempotent (re-creating a dismissed link revives it);
  // dismiss is a soft delete.
  await link.create(
    { module: "user", model: "user", id: u.id },
    { module: "organization", model: "organization", id: o.id },
  );
  await link.dismiss(
    { module: "user", model: "user", id: u.id },
    { module: "organization", model: "organization", id: o.id },
  );

  // fetch the linked module's full records (either direction)
  const orgs = await link.fetch(
    { module: "user", model: "user", id: u.id },
    { module: "organization", model: "organization" },
  );

  // nested cross-module graph query — Damat's analogue of query.graph
  const { data } = await link.graph({
    module: "user",
    entity: "user",
    fields: ["id", "email", "organizations.name", "organizations.slug"],
    filters: { id: u.id },
  });
  ```

  The service also exposes `list` (raw junction rows) and `listLinkedIds` (ids
  only). `graph` follows both cross-module links (via the junctions) and
  intra-module relations (via the owning service's `include`) to any depth,
  fetching per hop through each module's own service rather than issuing
  cross-module SQL joins.

## Added

- `defineLink(left, right, options?)` — build a link and its junction model.
- `defineLinkModule(links, id?)` — wrap a set of links as a `defineModule`-compatible
  instance registered as the `link` module.
- `collectLinkModels(links)` — the `models` map a links dir exports for
  migrate/codegen.
- `LinkService` via `getModule("link")` — `create` / `dismiss` / `list` /
  `listLinkedIds` / `fetch` / `graph`.
- `links` field in `damat.config.ts` (path or list of paths), picked up by both
  framework boot and the `damat-orm` CLI.
- Per-owner `link:<owner>` migration modules; auto-generated `<table>.links.ts`
  type augmentations woven into linked modules.
- Framework/CLI-internal helpers: `setLinkModuleResolver(fn)`,
  `resolveLinkModuleEntries(links, cwd)`, `resolveLinkMigrationModules(links, cwd)`,
  `renderLinkAugmentations(fields)`, plus naming helpers `defaultPivotTable` /
  `pivotColumns`.

## Action required

Links are entirely opt-in; existing apps that don't define any links are
unaffected. To start using links:

1. Add the dependency (it is published alongside the rest of the 0.1.3 line and
   re-exported by `@damatjs/framework` and `@damatjs/module`):

   ```bash
   bun add @damatjs/link
   ```

   App code imports `defineLink` / `collectLinkModels` / `defineLinkModule` from
   `@damatjs/framework`; a standalone module package imports the same names from
   `@damatjs/module`.

2. Create the links tree, mirroring a module's shape:

   ```
   src/links/
   ├── index.ts                       # defineLinkModule(...) for boot
   └── user/
       ├── models/user-organization.ts  # defineLink(...)
       ├── index.ts                      # links + collectLinkModels(links)
       └── migrations/
   ```

3. Point your config at it:

   ```ts
   // damat.config.ts
   export default defineConfig({ /* … */, links: "./src/links" });
   ```

4. Generate and apply the junction migration, then regenerate the linked
   modules' types:

   ```bash
   damat-orm migrate:create link:user
   damat-orm migrate:up
   damat-orm generate:types user
   damat-orm generate:types organization
   ```

5. Use it at runtime via `getModule("link")`.
