# @damatjs/link 0.6.0

> Links resolve against real tables and real primary keys: junction naming uses
> table names (not model keys), endpoint PKs come from the registry, pivot
> columns can be named explicitly, link creation is a race-safe upsert, and the
> graph API only serves entities that participate in registered links.

## What changed

### Real tables and primary keys

- `ResolvedEndpoint` gains a `table` field — resolved from the global model
  registry when loaded, else derived camelCase → snake_case. Junction names are
  built from **table names**, not model keys.
- An endpoint's `primaryKey` defaults to the model's actual PK from the registry
  (falling back to `id`); an explicit value still wins.
- With `database.foreignKeys: true`, pivot FK columns reference the real table
  and real PK column (e.g. `{ table: "lk_products", reference: "sku" }`) and
  cascade on delete.
- `pruneColumns` accepts a `primaryKey` parameter — a model keyed on `sku` keeps
  its real PK instead of getting a stray `id` forced in.

### Safer naming and creation

- Junction-name singularization is conservative: words ending in `ss`/`us`/`is`
  (address, class, status) are left intact instead of being mangled.
- `LinkOptions.pivotColumns` names the junction columns per side explicitly,
  e.g. `{ left: "member_id", right: "org_id" }`.
- `create` is a single `INSERT … ON CONFLICT` upsert — the old check-then-insert
  had a race window that produced duplicate-key errors and resurrected stale
  soft-deletes.

### Graph query guard

The graph API validates the root entity: only entities that participate in a
registered link can be queried, so the link service can't be used as a side door
into arbitrary modules.

## Upgrade notes

- Junction table names derived from model keys may differ once real table names
  are used — check generated pivot names before migrating production schemas
  (pin them with `pivotColumns`/explicit names where you need stability).
- Graph queries against entities with no registered links now fail — query the
  owning module's own API instead.
