# @damatjs/orm-cli 0.1.3

> The `damat-orm` CLI now understands cross-module links: it discovers
> `link:<owner>` migration modules from `damat.config.ts` and weaves linked
> fields into the linked modules' generated types.

## What changed

Before 0.1.3, the CLI only knew about the `modules` map in `damat.config.ts`.
Relationships between models in *different* modules had no first-class place in
the migrate/codegen pipelines.

0.1.3 teaches `loadModules` to also read a new `links` field in
`damat.config.ts`. The links tree mirrors modules
(`src/links/<owner>/{models,index.ts,migrations}`), and **each `<owner>`
directory becomes its own migration module** with id `link:<owner>` and
`kind: "link"`. These link modules carry the junction tables that join the
linked models.

**Migrations â€” before â†’ after.** Junction tables now migrate through the
*existing* commands; no new subcommand was added.

```ts
// damat.config.ts â€” after
export default {
  projectConfig: { databaseUrl: process.env.DATABASE_URL },
  modules: {
    user: { resolve: "./src/modules/user" },
    organization: { resolve: "./src/modules/organization" },
  },
  links: "./src/links", // NEW: each src/links/<owner> â†’ link:<owner>
};
```

```bash
# before: no way to migrate a cross-module junction table through the CLI

# after: the link owner is just another module id
bun damat-orm migrate:create link:user   # initial/diff migration for its junction tables
bun damat-orm migrate:status link:user   # applied vs pending for that link module
bun damat-orm migrate:list               # link:<owner> modules appear alongside real ones
bun damat-orm migrate:up                  # applies module + link migrations together
```

A `link:<owner>` id is only added when `src/links/<owner>` exists and exposes an
entry index; if it would collide with a real module id, the real module wins.
Apps with no `links` field behave exactly as before.

**Type generation â€” before â†’ after.** `generate:types` changed in two ways:

1. Running it for a **link module** is now a no-op â€” link modules don't emit
   their own (junction) types:

   ```bash
   bun damat-orm generate:types link:user
   # â†’ notice: "'link:user' is a link module; run generate:types for the linked
   #    modules to get the linked fields." (exit 0)
   ```

2. Running it for a **linked module** now augments that module's types with the
   linked entities. For every link the module participates in, the linked entity
   is added as an optional field on the module's interface via a sibling
   `<table>.links.ts` (declaration merging), re-exported from the types
   `index.ts`:

   ```bash
   bun damat-orm generate:types user
   # writes the usual model types PLUS users.links.ts adding e.g.
   #   organizations?: Organizations[]
   bun damat-orm generate:types organization
   # writes organizations.links.ts adding the reverse field
   ```

   Augmentation is best-effort: if a link directory can't be imported or a
   referenced table can't be resolved, the command logs a warning and writes the
   base model types anyway. With no link modules present, output is unchanged
   (model-only), so existing single-module projects see no difference.

Mechanically this is wired through the new `@damatjs/link` dependency
(`resolveLinkMigrationModules` for discovery, `renderLinkAugmentations` for the
`<table>.links.ts` content).

## Added
- `links` field support in `damat.config.ts`: discovers one `link:<owner>`
  migration module per `src/links/<owner>` directory.
- `generate:types` link augmentation â€” `<table>.links.ts` files added to linked
  modules' generated types and re-exported from their index.
- `@damatjs/link` added as a dependency.

## Changed / improved
- `migrate:create` / `migrate:up` / `migrate:status` / `migrate:list` now operate
  on `link:<owner>` modules with no special-casing â€” junction tables migrate
  through the same pipeline as regular modules.
- `generate:types <link:owner>` short-circuits with a notice instead of erroring
  or emitting empty output.
- Config cache-busting switched to a transient sidecar-file import (the config's
  module identity is no longer cached across loads).

## Breaking
- None. `links` is opt-in; projects without it are unaffected.

## Action required

Drop-in upgrade for existing projects. To adopt cross-module links:

1. Add a `links` field to `damat.config.ts` pointing at your links root (e.g.
   `links: "./src/links"`).
2. Create `src/links/<owner>/` with `models/`, an `index.ts` exporting your
   `links` (and a `migrations/` dir), one owner directory per link group.
3. Create and apply the junction-table migrations:

   ```bash
   bun damat-orm migrate:create link:<owner>
   bun damat-orm migrate:up
   ```

4. Regenerate types for the **linked modules** (not the link modules) to pick up
   the new fields:

   ```bash
   bun damat-orm generate:types <linked-module>
   ```
