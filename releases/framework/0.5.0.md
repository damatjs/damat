# @damatjs/framework 0.5.0

> Non-`Error` throws (`throw "boom"`, `throw { code: 1 }`) are no longer silently
> swallowed by the error middleware — they are stringified, logged, and surfaced
> in dev responses.

## What changed

`parseError` in the error middleware matched thrown values against `Error`
subclasses only. A route or service that threw a plain string or object fell
through every branch: the log carried an empty/generic entry and the response
gave no hint of what was thrown.

- **Before:** non-`Error` values matched no branch; their content was dropped.
- **After:** a final branch captures the value with a circular-safe
  `safeStringify(value)`, logs it as `Unhandled non-error throw`, and (in dev
  mode) includes the stringified value in the response message.

Also in this version, from the 0.5.0 codebase audit:

- `loadConfigAsync` now uses Node's built-in `url.pathToFileURL` (with an explicit
  `path.resolve`) instead of a hand-rolled converter — same behavior, better
  cross-platform footing.
- `initializeServices` starts `healthChecks` as an empty object; probes are added
  only for services that are actually configured (previously placeholder stubs
  were always created and then overwritten).

## Upgrade notes

Nothing to do. Errors that were logged before are logged the same way; throws
that were invisible before are now visible.
