# @damatjs/workflow-engine 0.3.6

> Steps now return a `StepResponse(output, compensateInput?)` and compensation receives `(compensateInput, ctx)` — a typed, explicit way to carry rollback data that the old `(input, output)` signature couldn't express.

## What changed

A step's forward function used to return its output directly, and its
compensation was called with `(input, output, ctx)`. That works when the rollback
data *is* the output (create/delete), but it can't carry anything else — an
`update` step needs the **prior** row, which is neither the input nor the
post-update output, so apps reached for side channels.

Now `invoke` returns a `StepResponse` that carries the downstream `output` **and**
a separate `compensateInput`, and `compensate` receives **only** that payload
(plus context).

```ts
// Before
createStep<UpdateInput, Row>(
  "update",
  async (input, ctx) => {
    const [row] = await svc.update({ where: { id: input.id }, data: input.data });
    return row;                                  // prior row is gone
  },
  async (input, output, ctx) => { /* output is the UPDATED row — can't restore */ },
);

// After
createStep<UpdateInput, Row, Row>(
  "update",
  async (input, ctx) => {
    const previous = await svc.find({ where: { id: input.id } });
    const [row] = await svc.update({ where: { id: input.id }, data: input.data });
    return new StepResponse(row, previous);      // output downstream; previous = rollback payload
  },
  async (previous, ctx) => {
    await svc.update({ where: { id: previous.id }, data: { ...previous } });
  },
);
```

There is **no output fallback**: a step that provides no `compensateInput` gives
its compensation `undefined`, never the output. The payload is **type-enforced** —
when the third generic `C` excludes `undefined`, omitting it is a compile error;
opt out by typing `C` as `… | undefined`.

## Added
- `StepResponse<O, C>` value export — `new StepResponse(output, compensateInput?)`
  plus the `StepResponse.isStepResponse(...)` guard.
- A third generic `C` (compensation payload type, default `undefined`) on
  `createStep`, `executeStep`, `StepDefinition`, and the control-flow helpers
  (`runStep`, `when`, `ifElse`).

## Changed / improved
- `invoke` returns `Promise<StepResponse<O, C>>`; `executeStep` unwraps it once so
  the workflow and downstream steps still receive the plain `O`.
- `StepResponse` is detected by a realm-global `Symbol.for` brand (not
  `instanceof`), so it can't be misclassified across duplicate copies of the
  package.

## Breaking
- **Compensation signature** changed from `compensate(input, output, ctx)` to
  `compensate(compensateInput, ctx)`. It no longer receives the input or output —
  put whatever rollback needs into the `StepResponse`'s second argument.
- **`invoke` must return a `StepResponse`.** A bare value no longer type-checks
  (the engine still tolerates one at runtime, treating it as the output with no
  payload, but the types require the wrapper).
- **No output fallback.** Where compensation previously received the output, it
  now receives `undefined` unless a `compensateInput` was provided.

## Action required

1. **Wrap step returns** in `StepResponse`:
   ```ts
   - async (input, ctx) => doWork(input)
   + async (input, ctx) => new StepResponse(await doWork(input))
   ```
2. **Move rollback data into the payload and update the compensation signature.**
   If your compensation used the output, pass it explicitly and add the third
   generic when the payload type differs from `O`:
   ```ts
   - createStep<I, Row>(name,
   -   async (input, ctx) => create(input),
   -   async (input, created, ctx) => remove(created.id));
   + createStep<I, Row, string>(name,
   +   async (input, ctx) => { const r = await create(input); return new StepResponse(r, r.id); },
   +   async (id, ctx) => remove(id));
   ```
3. **Re-generate or migrate CRUD steps.** Apps that regenerate via
   `@damatjs/codegen` pick up the new shape automatically; hand-written steps must
   be updated.
