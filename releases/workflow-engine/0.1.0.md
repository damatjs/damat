# @damatjs/workflow-engine 0.1.0

> Stabilization minor: workflow-level step-config layering, AbortSignal cancellation, true per-attempt timeouts, a capped exponential backoff, `compensationsFailed` tracking, and retry exhaustion surfaced as `MAX_RETRIES_EXCEEDED` on the workflow result.

## What changed

`0.1.0` keeps the same public API — `createStep`, `createWorkflow`,
`executeStep`, the control-flow helpers, and the lock primitives are all called
exactly as before — but hardens the execution engine in `src/step/execute.ts` and
the result mapping in `src/workflow/execute.ts`. The observable changes:

### 1. Per-attempt timeout (was: one timeout around all retries)

Before, the step `timeoutMs` wrapped the **entire retried effect** — the timeout
budget covered the first attempt *plus* every retry and the delays between them.

```ts
// Before (0.0.2): timeout wraps the whole retry loop
let executionEffect = Effect.tryPromise({ try: () => step.invoke(input, ctx) });
if (shouldRetry) executionEffect = Effect.retry(executionEffect, { schedule, while });
const timedEffect = Effect.timeoutFail(executionEffect, { duration: timeoutMs, ... });
```

Now `timeoutMs` is applied **per attempt**: each individual invocation is bounded,
and a timed-out attempt becomes a retryable `StepTimeoutError`.

```ts
// After (0.1.0): timeout wraps a single attempt; retry wraps the timed attempt
const attemptEffect = Effect.timeoutFail(
  Effect.tryPromise({ try: (signal) => step.invoke(input, { ...ctx, attempt }, signal) }),
  { duration: timeoutMs, onTimeout: () => new StepTimeoutError(...) },
);
if (shouldRetry) executionEffect = Effect.retry(attemptEffect, { schedule, while });
```

### 2. AbortSignal forwarded to `invoke`

`invoke` now receives a third argument — the Effect-provided `AbortSignal` —
`(input, ctx, signal) => Promise<O>`. Forward it into `fetch`/db calls so a
timeout actually cancels in-flight work (JS can't kill a bare promise). Before,
`invoke` was called with only `(input, ctx)`.

### 3. `isRetryable` receives the *original* error (was: the engine wrapper)

Before, the retry predicate was handed the `StepExecutionError` wrapper, and the
default Validation check reached into `error.cause`. Now the predicate is handed
the **original** error your `invoke` threw (unwrapped from `StepExecutionError`),
or the `StepTimeoutError` for timeouts.

```ts
// After (0.1.0)
while: (error) => {
  const original = error instanceof StepExecutionError ? (error.cause ?? error) : error;
  return retryPolicy.isRetryable
    ? retryPolicy.isRetryable(original)
    : !(original instanceof Error && original.name === "ValidationError");
}
```

### 4. `maxDelayMs` is now a flat per-delay cap

Before, `maxDelayMs` was enforced with `Schedule.whileOutput(d => d <= maxDelayMs)`
— retrying simply **stopped** once a computed delay exceeded the cap. Now the
schedule is `exponential ∪ spaced(maxDelayMs)` intersected with `recurs(maxAttempts)`,
so each delay is **capped at** `maxDelayMs` and the full `maxAttempts` retries
still run.

### 5. Retry exhaustion surfaced at the workflow boundary

Before, when retries were exhausted the step failed with a
`MaxRetriesExceededError` (and `executeStep`'s error channel included it). Now the
step **re-fails with its last `StepExecutionError`/`StepTimeoutError`**, and the
engine records a `MaxRetriesExceededError` on `ctx.engineState.retriesExceeded`.
`executeWorkflowInternal` promotes that recorded error to `result.error`, so
callers still see `result.error.code === "MAX_RETRIES_EXCEEDED"`. The step-level
error channel is now just `StepExecutionError | StepTimeoutError`.

### 6. `compensationsFailed` tracking + accurate `ctx.attempt`

A `WorkflowFailure` now reports `compensationsFailed` (compensations that threw —
logged and swallowed, never raised) alongside `compensated`. And the `attempt`
your `invoke` receives is the live attempt counter (1-based), incremented on each
real invocation, instead of the static `ctx.attempt`.

## Added
- AbortSignal (3rd arg) passed to step `invoke`.
- `compensationsFailed` on `WorkflowFailure`.
- `WorkflowEngineState.retriesExceeded` (internal) and workflow-boundary promotion
  to `MAX_RETRIES_EXCEEDED`.
- Workflow-level `defaultStepConfig` layering applied in `resolveStepConfig`
  (engine defaults < workflow `defaultStepConfig` < the step's own `rawConfig`),
  with `createStep` retaining the unmerged `rawConfig` for this layering.

## Changed / improved
- `timeoutMs` is per attempt, not per step run.
- `maxDelayMs` caps each backoff delay (via `Schedule.union(spaced)`) instead of
  terminating retries early.
- `isRetryable` is handed the original (unwrapped) error.
- `MAX_RETRIES_EXCEEDED` is surfaced on the workflow result rather than from the
  step's error channel.

## Breaking

These are behavioral changes within an unchanged API surface. Treat them as
breaking if you relied on the old behavior:

- **`isRetryable` argument shape.** A predicate that inspected the
  `StepExecutionError` wrapper (e.g. `e.cause`, `e.code`) now receives the original
  thrown error directly. Rewrite it to test the original error.
- **Per-attempt timeout.** A `timeoutMs` previously sized to cover *all* attempts
  will now apply to each attempt individually. Re-tune `timeoutMs` (per attempt)
  and bound total runtime with the workflow `timeoutMs`.
- **`executeStep` error channel narrowed** from
  `StepExecutionError | StepTimeoutError | MaxRetriesExceededError` to
  `StepExecutionError | StepTimeoutError`. Code that matched
  `MaxRetriesExceededError` on a raw step effect must now read it from
  `result.error` after the workflow runs (`result.error.code === "MAX_RETRIES_EXCEEDED"`).

## Action required

1. **Forward the AbortSignal** in steps that do cancelable I/O so timeouts take
   effect:
   ```ts
   - createStep("call", async (input, ctx) => fetch(url))
   + createStep("call", async (input, ctx, signal) => fetch(url, { signal }))
   ```
2. **Re-check `timeoutMs`** — it is now per attempt. If a step retries, total time
   is roughly `(attempts × timeoutMs) + backoff`; cap it with the workflow
   `timeoutMs` if needed.
3. **Update `isRetryable` predicates** to test the original error rather than the
   `StepExecutionError` wrapper.
4. **Read `MAX_RETRIES_EXCEEDED` from the workflow result**, not the step effect:
   ```ts
   const result = await workflow.execute(input);
   if (!result.success && result.error.code === "MAX_RETRIES_EXCEEDED") { /* ... */ }
   ```
