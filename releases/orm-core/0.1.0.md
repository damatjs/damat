# @damatjs/orm-core 0.1.0

> First published minor release: the database-agnostic runtime layer (`ModelRegistry` + `QueryLogger`) for the Damat ORM, stabilized for general use, with a relation-resolution correctness fix and a unit-test suite.

## What changed

`0.1.0` is the baseline minor release of `@damatjs/orm-core`. It is the thin
runtime seam between schema definitions (`@damatjs/orm-model`) and the database
driver (`@damatjs/orm-pg`): it contains no SQL and no connection logic. It
provides two facilities every driver needs:

- **`ModelRegistry`** — indexes `ModelDefinition`s by logical name and by table
  name, snapshots each model's column list at registration, and resolves relation
  targets. Drivers build one, fill it from the app's models, and consult it during
  query construction.
- **`QueryLogger`** — a structured logger for queries, slow queries, errors, and
  transaction boundaries, each category independently switchable, available as a
  process-wide singleton so every driver shares one logging configuration.

At this baseline the package exports `ModelRegistry`, `ModelRegistryError`,
`QueryLogger`, the singleton helpers `getQueryLogger` / `setQueryLogger` /
`configureQueryLogger`, and the `QueryLoggerOptions` / `ModelRegistryEntry` types.

Two concrete improvements land in this release versus the pre-release code:

**Relation resolution now matches on the property name.** `resolveRelation`
previously found a relation by testing whether the property name was a _key of the
`RelationSchema` object_:

```ts
// before
const relation = schema.relations?.find((r) => propertyName in r);
```

That only matched when the caller happened to pass one of the schema's own field
names (`from`, `to`, `type`, …), so resolving a real relation by its property name
generally returned `undefined`. It now compares against the property the relation
was defined under:

```ts
// after
const relation = schema.relations?.find((r) => r.from === propertyName);
```

`resolveRelation(modelName, propertyName)` therefore resolves a model's actual
relations and returns the target entry via `getByTableName(relation.to)`.

**A unit-test suite was added.** The `test` script changed from `exit 0` to
`bun test`, and `src/__tests__/` now holds `registry.test.ts`, `logger.test.ts`,
`index.test.ts`, and a shared `helpers.ts`, covering the registry's name/table
indexes and relation resolution and the logger's category gating and singleton
helpers.

## Added

- `ModelRegistry` with `register` / `registerMany`, `get` / `getByTableName`,
  `getColumns`, `getAll`, `has`, `getTableNames` / `getModelNames`, and
  `resolveRelation`.
- `ModelRegistryError` — a named error for _consumers_ to throw on a
  required-but-missing model (the registry's own getters return `undefined`).
- `QueryLogger` with `logQuery` / `logQueryError` / `logSlowQuery` /
  `logTransaction`, runtime controls `setOptions` / `enable` / `disable`, and the
  `getQueryLogger` / `setQueryLogger` / `configureQueryLogger` global-singleton
  helpers.
- `QueryLoggerOptions` and `ModelRegistryEntry` type exports.
- A `bun test` unit-test suite under `src/__tests__/`.

## Changed / improved

- `ModelRegistry.resolveRelation` now matches relations by `r.from ===
propertyName` instead of `propertyName in r`, so resolving a relation by its
  property name works as intended.

## Breaking

- None. `resolveRelation` previously returned `undefined` for normal property-name
  lookups, so the fix only makes it start returning results that callers expected;
  no working call changes meaning.

## Action required

None — drop-in. New consumers add the dependency and import from the package root:

```bash
bun add @damatjs/orm-core
```

In practice you rarely import this package yourself; the Postgres driver
(`@damatjs/orm-pg`) and the framework wire it up.
