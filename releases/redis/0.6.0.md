# @damatjs/redis 0.6.0

> Every hot path becomes atomic or non-blocking: `SCAN`-based pattern deletion,
> Lua-scripted rate limiting and dequeue, TTL applied only on a counter's first
> increment, and capped completed/failed queue sets.

## What changed

### Cache

- **`cacheDeletePattern`** used blocking `KEYS` — O(N) over the whole keyspace,
  stalling every client on large databases. It now walks a cursor-driven `SCAN`
  loop (`COUNT 100`) and deletes in batches.

### Counters

- **`incrementCounter`** refreshed the TTL on every increment, so steady traffic
  could keep an expired window alive forever. The Lua script now applies the TTL
  **only on the first increment**; the window expires on schedule regardless of
  traffic.

### Queues

- **Dequeue is atomic.** The multi-step pipeline (read, claim, move) is replaced
  by a single Lua script — two workers can no longer claim the same job.
- **Terminal sets are capped.** `:completed` and `:failed` sets trim themselves
  to `DEFAULT_MAX_TERMINAL_ENTRIES` (10 000). The constructor accepts
  `RedisQueueOptions` to tune visibility timeout and the completed/failed caps.

### Rate limiting

- **`checkRateLimit` / `checkMultiRateLimit`** pruned, counted, and recorded in
  separate round-trips — a rejected request still wrote to the window and
  extended the key TTL, so a flood could hold a window open. Both now run as
  single Lua scripts: prune + count + record happen atomically, and **rejected
  requests record nothing**.
- **`checkMultiRateLimit` is all-or-nothing:** an entry is recorded only when
  every window passes.

## Upgrade notes

APIs are unchanged; behavior tightens. If you relied on rejected requests
extending a rate-limit window (unlikely, but it was the old behavior), that no
longer happens. Queue constructors accept optional `RedisQueueOptions` — omit it
to keep the defaults.
