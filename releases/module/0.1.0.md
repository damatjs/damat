# @damatjs/module 0.1.0

> First minor release: the whole Damat module system gathered into one package — author a module, run it as a live app, test it standalone against Postgres, and address/validate it for a registry, all from a single import.

## What changed

This is the package's first-of-its-kind release. Earlier `0.0.x` tags were
pre-alpha build-and-packaging iterations; `0.1.0` is the first version stabilized
for use, with expanded test coverage and known issues resolved.

`@damatjs/module` makes a Damat *module* — a self-contained vertical slice of
models + migrations + service + config + workflows + routes — developable,
runnable, and distributable on its own, before it is ever installed into a
backend app. It is the single direct dependency a module package declares; it
re-exports everything underneath so module code is insulated from the versions of
the sibling packages. The package is organized by concern, each re-exported from
the root barrel (`src/index.ts`):

- **Authoring** (`src/authoring.ts`) — one import for everything a module needs:
  `defineModule` / `ModuleService` (from `@damatjs/services`), `model` / `columns`
  (from `@damatjs/orm-model`), the workflow engine (`createStep` /
  `createWorkflow` / `executeStep` / `runStep` / `skipStep` / `parallel` / `when`
  / `ifElse` / `RetryPolicies` / `Effect`), app-side registry access (`getModule`
  / `hasModule` / `registerModule`), route contracts (`RouteHandler` /
  `RouteValidator`), and `z` (Zod). A module's `package.json` therefore depends on
  `@damatjs/module` only.

- **Manifest** (`src/manifest/`) — the `module.json` contract that makes a module
  portable. `ModuleManifest` (`name`, `version?`, `description?`, `author?`,
  `env?`, `packages?`, `modules?`, `paths?`, `registry?`) plus
  `validateModuleManifest` (hand-rolled, CLI-friendly errors; enforces kebab-case
  `name`; permissive about unknown keys), `readModuleManifest`,
  `MODULE_MANIFEST_FILENAME = "module.json"`, and `DEFAULT_MODULE_PATHS` (the
  standard `index.ts` / `models` / `migrations` / `workflows` / `types` layout).

- **Config** (`src/config/`) — `module.config.ts`, the only thing a module author
  configures for the standalone runtime. `defineModuleConfig` (a type-safe
  identity helper), `loadModuleConfig` (dynamic-imports the file; returns `{}`
  when absent), and the `ModuleAppConfig` type. Database/redis always come from
  `DATABASE_URL` / `REDIS_URL`, never from this file.

- **Harness** (`src/harness/`) — boot one module **without an HTTP server** for
  dev and test. `bootModule` wires `ConnectionManager` + `PoolManager`, applies
  the module's own migrations, calls `module.init()`, and hands back the service;
  `withModule` boots → runs → always tears down. Requires a real Postgres.

- **Runtime** (`src/runtime/`) — run one module package as a **live HTTP app**
  with the framework's full stack (middleware, file routes, health checks).
  `startModuleApp` (what `damat module dev` boots; `port: 0` for an ephemeral port
  in API tests), `runModuleEntry`, `buildModuleAppConfig`, `DEFAULT_MODULE_PORT =
  7654`, and `locateModuleDir` (finds `module.json` in `src/` or the package
  root).

- **Tooling** (`src/tooling/`) — `createModuleMigration` (diff the module's models
  against the last snapshot) and `generateModuleTypes` (row types + zod), both
  operating on a standalone package with **no** `damat.config.ts`.

- **Registry** (`src/registry/`) — how modules are addressed, distributed, and
  trusted. `parseModuleRef` / `formatModuleRef` (ref grammar
  `namespace?/name@version?`), the `RegistryIndex` / `RegistryModuleEntry` schema
  with `normalizeVersionEntry`, `validateModuleDir` (errors block install,
  warnings block publish), and the install-time trust gate `verificationPolicy` /
  `evaluateVerification` (`DAMAT_MODULE_VERIFY` / `DAMAT_MODULE_REGISTRY`).
  `resolveRegistryEntry` / `resolveRegistryRef` resolve a ref against an index.

The same module directory is consumed three ways from these concerns: standalone
(harness, no HTTP), live app (runtime, full HTTP stack), and distribution
(registry resolution + the verification gate).

## Added

- Authoring barrel (`defineModule`, `ModuleService`, `model`, `columns`, the
  workflow engine, `getModule` / `hasModule` / `registerModule`, route types,
  `z`) — the single import surface for module packages.
- Manifest contract: `ModuleManifest` & friends, `validateModuleManifest`,
  `readModuleManifest`, `MODULE_MANIFEST_FILENAME`, `DEFAULT_MODULE_PATHS`.
- Module config: `defineModuleConfig`, `loadModuleConfig`, `ModuleAppConfig`.
- Standalone harness: `bootModule`, `withModule`, `BootModuleOptions` /
  `BootedModule` / `BootableModule`.
- Module-as-app runtime: `startModuleApp`, `runModuleEntry`,
  `buildModuleAppConfig`, `DEFAULT_MODULE_PORT`, `locateModuleDir`,
  `StartModuleAppOptions` / `RunningModuleApp`.
- Authoring tooling: `createModuleMigration`, `generateModuleTypes`.
- Registry: `parseModuleRef`, `formatModuleRef`, `validateModuleDir`,
  `resolveRegistryEntry`, `resolveRegistryRef`, `verificationPolicy`,
  `evaluateVerification`, `normalizeVersionEntry`, `VERIFICATION_STATUSES`, and
  the registry/verification types.

## Changed / improved

- Core functionality stabilized and test coverage expanded relative to the
  `0.0.x` pre-alpha line.

## Breaking

- None — this is the baseline minor release.

## Action required

None — drop-in for new module packages. Add `@damatjs/module` as the package's
only direct Damat dependency, place a `module.json` next to an `index.ts` that
`export default defineModule(...)`, and import everything else from
`@damatjs/module`. Develop/test with `bootModule` / `withModule`, run with
`startModuleApp` (or `damat module dev`).
