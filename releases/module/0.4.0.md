# @damatjs/module 0.4.0

> New tooling functions `runModuleMigration` and `runModuleMigrationStatus` apply
> and report a standalone module's own migrations against `DATABASE_URL` — no
> `damat.config.ts`, scoped to the single module. These back the new
> `damat module migration:run` / `migration:status` CLI commands.

## What changed

The tooling surface previously offered `createModuleMigration` (diff models → SQL)
and `generateModuleTypes`, both of which work offline. Applying a module's
migrations against a database was only reachable through the dev/test harness
(`bootModule`, `startModuleApp` → `applyModuleMigrations`).

Two new exported functions make the apply/inspect path available directly, the
same way `createModuleMigration` exposes the create path:

```ts
import { runModuleMigration, runModuleMigrationStatus } from "@damatjs/module";

const run = await runModuleMigration(process.cwd());
// { moduleName, applied: string[], pending: string[], success, error?, hadMigrations }

const status = await runModuleMigrationStatus(process.cwd());
// { moduleName, applied: number, pending: number, migrations: {name, applied}[], hadMigrations }
```

Both locate the module dir and read its manifest, resolve `DATABASE_URL` via the
same wiring `bootModule` uses (`resolveDatabaseConfig` → `ConnectionManager`), and
operate on a **single-module container** so only this module's migrations are
touched. Migrations are tracked under the module's name, so `runModuleMigration`
is idempotent and `runModuleMigrationStatus` reads its counts from the same key.
When the module has no migrations directory yet, both return early with
`hadMigrations: false` instead of connecting.

## Added

- `runModuleMigration(packageDir)` — apply a standalone module's own migrations to
  `DATABASE_URL`; returns applied/pending names + success.
- `runModuleMigrationStatus(packageDir)` — report a module's applied vs pending
  migrations without changing anything.
- Result types `RunModuleMigrationResult`, `ModuleMigrationStatusResult`,
  `ModuleMigrationStatusEntry`.

## Changed / improved

- Tooling docs clarify which functions touch the database (`runModuleMigration` /
  `runModuleMigrationStatus`) vs which are offline (`createModuleMigration` /
  `generateModuleTypes`).

## Breaking

- None. Existing exports (`createModuleMigration`, `generateModuleTypes`,
  `applyModuleMigrations`) are unchanged.

## Action required

None — drop-in, additive. Call the new functions where you need a module's
migrations applied/inspected without a full app boot.
