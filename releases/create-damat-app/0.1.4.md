# @damatjs/create-damat-app 0.1.4

> `--module` now scaffolds a module **locally** via `damat module init` instead of
> cloning a remote starter, and `create-damat-app <name>` works without typing
> `create`.

## What changed

### `--module` scaffolds locally

`create-damat-app <name> --module` previously tried to clone a remote module starter
repo, which did not reliably exist. It now scaffolds the module **locally** by
shelling out to `@damatjs/damat-cli module init <name>` — the same deterministic,
offline output — then installs dependencies on top. Pass `--repo-url <git>` to clone
a custom starter repo instead.

```bash
create-damat-app my-module --module                  # local scaffold (damat module init)
create-damat-app my-module --module --repo-url <git> # clone a custom starter
```

The **app** path is unchanged: `create-damat-app <name>` (no `--module`) clones the
default starter (`damat-starter-default`) and re-inits git.

### Default command

`create-damat-app <name>` now runs without the explicit `create` verb, via the new
`defaultCommand` support in [`@damatjs/cli` 0.1.4](../cli/0.1.4.md).

## Added

- `--repo-url <git>` to clone a custom starter (apps or modules).
- Local module scaffolding that wraps `damat module init`.

## Changed / improved

- `--module` no longer requires a reachable remote starter repo.

## Breaking

- None. The app flow and flags are unchanged.

## Action required

- None — drop-in.
