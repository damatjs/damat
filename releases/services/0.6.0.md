# @damatjs/services 0.6.0

> Request-derived query options are sanitized end to end: only safe find options
> pass through, `orderBy` is whitelist-validated, pagination is bounded, reads
> filter soft-deleted rows by default, and `update`/`updateOne` stamp
> `updated_at` automatically.

## What changed

### Find-option sanitization

A new `buildFindOptions` step filters options coming from a request down to the
safe set — `select`, `where`, `orderBy`, `take`, `skip`, `include`. Dangerous
keys that used to pass straight through (`whereRaw`, `allowFullTable`,
`distinct`, …) are dropped.

### `orderBy` validation

`sanitizeOrderBy` upper-cases and whitelist-checks direction (`ASC`/`DESC`) and
the new `nulls` field (`NULLS FIRST`/`NULLS LAST`) — mirroring the same guard in
`@damatjs/orm-pg` 0.6.0.

### Bounded pagination

`take` is capped at `MAX_PAGE_SIZE` (1000) and `skip` must be a non-negative
integer; out-of-range values are rejected instead of forwarded.

### Soft-delete aware reads

`find`, `count`, and `exists` automatically add `deleted_at IS NULL` unless the
caller passes `withDeleted: true` (or is already filtering on `deleted_at`).
Previously soft-deleted rows appeared in every read that forgot the filter.

### `updated_at` stamping

`update` and `updateOne` set `updated_at` to the current time unless the caller
provided a value explicitly.

### Narrowed delete contract

`delete` forwards only `where` and `returning` — an untrusted payload can no
longer smuggle `cascade` or `allowFullTable` into the call.

## Upgrade notes

- Reads that relied on seeing soft-deleted rows must now pass
  `withDeleted: true`.
- Requests sending `take` above 1000 (or a negative `skip`) now fail validation.
- If a request payload was (deliberately or not) driving `cascade`/`whereRaw`
  through service calls, that path is closed — call those options from code, not
  from request data.
