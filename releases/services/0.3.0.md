# @damatjs/services 0.3.0

> Adds `upsert`/`upsertMany`, cascade delete, and row-returning `updateOne` / `findById` / `findOne` to the generated per-model accessor (`ModelMethods`). Additive — every existing call keeps its behaviour.

## What changed

`ModuleService({ models })` accessors gain five new methods plus an opt-in
cascade flag on the two delete methods. Everything is usable inside
`this.transaction(cb)` and fully typed, so `getModule("<name>").<table>.…` keeps
its types.

### Upsert

- **`upsert(options)` → `Promise<T>`** and **`upsertMany(options)` → `Promise<T[]>`**
  wrap the ORM's insert-on-conflict support. Options mirror the ORM layer:
  `{ data, onConflict, updateColumns?, set?, returning? }` (a single row for
  `upsert`, an array for `upsertMany`). `data` is validated as a full row before
  the write.

### Cascade delete

- **`delete({ where, cascade: true })`** and **`softDelete({ where, cascade: true })`**.
  `cascade` defaults to `false`, so current callers are unchanged. When set, the
  call runs in a transaction (reusing an open one, otherwise opening its own) and,
  for every `hasMany`/`hasOne` relation on the model, removes the child rows whose
  FK references the matched rows — depth-first and recursively — before removing
  the matched rows themselves.

  Each relation's `rule.onDelete` is honoured: no rule / `CASCADE` recurses;
  `SET NULL` nulls the child FK; `RESTRICT` / `NO ACTION` throw when children
  exist. Relation cycles are broken with a visited-set, and any error rolls the
  whole transaction back. `delete` returns the total rows removed across all
  levels; `softDelete` returns the soft-deleted top-level rows.

  This replaces hand-rolled, per-module transactional cascade-deletes with one
  relation-aware implementation.

### Row-returning writes

- **`updateOne(options)` → `Promise<T | null>`** updates and returns the single
  affected row (or `null`), removing the update-then-find re-read that `update`'s
  `T[]` return otherwise forces.
- **`findById(id, options?)`** and **`findOne(where, options?)`** are thin
  conveniences over `find`; both return `T | null` and accept the usual
  `select` / `include` options.

## Upgrade notes

Nothing to do. All additions are new methods or a new optional `cascade` flag
that defaults to `false`. `delete` / `softDelete` without `cascade` behave exactly
as before.

Cascade delete relies on the model's relation metadata
(`toTableSchema().relations`) and resolves each child's FK column the same way
relation loading does (`linkedBy`, else `<mappedBy>_id`, else `<modelName>_id`).
Models reached through a cascaded relation must be registered with the same
entity manager (they are, when they belong to the same `ModuleService`).

Pairs with `@damatjs/orm-pg` 0.3.0, which adds the `upsertMany` execution path
this release plumbs through.
