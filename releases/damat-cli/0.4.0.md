# @damatjs/damat-cli 0.4.0

> A module author can now apply and inspect a module's own migrations against a
> real database from the CLI — two new commands, `damat module migration:run` and
> `damat module migration:status`, complete the per-module migrate loop alongside
> the existing `migration:create`.

## What changed

Until now a standalone module could **create** a migration
(`damat module migration:create` diffs models → SQL) but had no CLI way to
**apply** it without booting the module as a full app (`damat module dev`). To
test a module against a real Postgres you had to start the server or write a
`bootModule` harness test.

Two commands close that gap, both scoped to the current module package and
reading `DATABASE_URL` from its `.env` (no `damat.config.ts`):

```bash
damat module migration:run      # apply this module's pending migrations
damat module migration:status   # report applied vs pending for this module
```

`migration:run` connects, applies only this module's own migration files —
tracked under the module's name, so re-running is idempotent — and disconnects.
It prints the applied migration names, or `No pending migrations`, or hints to
run `migration:create` first when the module has none. `migration:status`
reports a `<module>: N applied, M pending` headline plus a per-migration list,
without changing anything.

The scaffold (`damat module init`) and its `AGENTS.md` / `README.md` now include
the matching `migration:run` and `migration:status` npm scripts and document the
migrate-and-test loop.

## Added

- `damat module migration:run` — apply the current module's migrations to
  `DATABASE_URL` (scoped to the module; idempotent).
- `damat module migration:status` — show the current module's applied vs pending
  migrations.
- Scaffolded modules ship `migration:run` / `migration:status` package scripts.

## Changed / improved

- The module command group help, CLI reference, and authoring guide list the full
  per-module migrate trio (`create` / `run` / `status`).

## Breaking

- None.

## Action required

None — drop-in. Inside a module package, set `DATABASE_URL` in `.env`, then
`damat module migration:create` → `damat module migration:run`; use
`damat module migration:status` to check state. Existing scaffolded modules can
add the two scripts (or just call the commands directly).
