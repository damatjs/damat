# @damatjs/damat-cli 0.1.0

> First minor release of `damat` — the user-facing CLI for developing, building,
> and composing Damat backends and modules.

## What changed

This is the package's baseline release. `@damatjs/damat-cli` ships the `damat`
binary: a thin command layer over the shared `@damatjs/cli` runner, the
`@damatjs/module` runtime/registry library, and `@damatjs/framework`. The CLI is
mostly orchestration — the real module logic (manifest read/validate, registry
resolve/verify, per-module migration/codegen, the module dev entry) lives in
`@damatjs/module`.

It provides two layers of commands:

- **App lifecycle** — `dev` (alias `d`), `build` (alias `b`), and `start`
  (alias `s`). `dev` writes a generated `runEntry()` file under `.damat/` and
  runs `bun --watch`; `build` bundles `.damat/dist/entry.js` with `bun build`,
  then copies the project's `src/` and builds `damat.config.ts` into the output;
  `start` runs the built `entry.js` (erroring if not built). `dev` defaults
  `NODE_ENV` to `development`, `start` to `production`, both loading env via
  `@damatjs/load-env`.
- **`module` group** (alias `m`) — a shadcn-style toolchain. App-side: `add`
  (install a module from a registry ref, local path, or git — copy source into
  `src/modules/<id>`, register it in `damat.config.ts` with provenance, sync env
  into `.env.example`, `bun add` its packages, and gate registry installs through
  a verification check) and `list`/`ls` (list installed modules + provenance).
  Authoring side, all operating on the current module package: `init <name>`
  (scaffold a runnable standalone package), `dev` (run the package standalone),
  `migration:create` (diff models → migration), `codegen` (row types + zod
  schemas), and `validate` (contract + registry-readiness check).

App-lifecycle commands operate on the current project (`ctx.cwd`); authoring
commands operate on the current module package; `module add` is the one
cross-cutting command that mutates the surrounding app. Every handler returns an
explicit `{ exitCode }` and reports through `ctx.logger`.

## Added

- `damat` binary with `dev`, `build`, `start`, and the `module` command group
  (`add`, `list`, `init`, `dev`, `migration:create`, `codegen`, `validate`).
- Library entry that re-exports `@damatjs/cli` (command types + `runCli`).

## Changed / improved

- Stabilized core functionality, expanded test coverage, and resolved known
  issues relative to the `0.0.x` pre-alpha line.

## Breaking

- None — first stable minor.

## Action required

None — drop-in for new projects. In a generated project `damat` is already a
devDependency; invoke it via Bun:

```bash
bun damat dev                 # dev server, hot reload
bun damat build               # production build
bun damat start               # run the build
bun damat module add user     # install a module
```
