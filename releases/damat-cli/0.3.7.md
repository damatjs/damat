# @damatjs/damat-cli 0.3.7

> A module can ship cross-module link files (real `defineLink`s); `damat module add`
> now splits them into `src/links/<moduleId>/` the same way it splits routes and
> workflows. The migration stays on the backend, so a shipped link is dormant until
> activated ā€” no drafts, no `link-setup` command.

## What changed

Installing a module-declared link used to be a five-step dance: declare a
blank-target `link[]` rule in `module.json`, let `add` seed
`src/links/.link-drafts.json`, fill the target, run `damat module link-setup` to
materialize a `defineLink`, then migrate. That was too many steps.

Now a module **ships the link as a real `defineLink` file** under `links/models/`,
authored exactly the way an app authors `src/links/`. On `damat module add` it is
split into the app like any other layer:

```
links/models/<from>-<to>.ts   ā†’   src/links/<moduleId>/models/<from>-<to>.ts
```

After copying, `add`:

- regenerates the owner `src/links/<moduleId>/index.ts` (a `collectLinkModels`
  barrel) and the top-level `src/links/index.ts` aggregator (a `defineLinkModule`)
  from the filesystem ā€” additive and idempotent, so other owners and hand-authored
  links are preserved;
- ensures `links: "./src/links"` in `damat.config.ts`;
- creates an empty `src/links/<moduleId>/migrations/`.

The module ships **no link migration** ā€” `add` prints the activation steps
(`damat-orm migrate:create link:<moduleId>` ā†’ `migrate:up` ā†’ `damat codegen`) but
does not run them. Until you do, the link is **dormant**; and because the link
runtime resolves endpoints lazily, a shipped link whose target module isn't
installed is **inert until queried**, never a boot error. Link model files are
copied **skip-existing** (overwritten only with `--force`), so you can edit a copied
link's target (e.g. when the target module was installed under a different id)
without a re-install clobbering it.

## Added

- `damat module add` splits a module's `links/models/*.ts` into
  `src/links/<moduleId>/`, regenerates the owner index + top-level aggregator, and
  ensures `links:` in `damat.config.ts`.
- The post-install "Next steps" list the link activation commands when the module
  shipped links.

## Changed / improved

- Link model files are copied skip-existing (overwritten only with `--force`) to
  preserve owner edits.

## Breaking

- The previous draft-based flow is removed: the `src/links/.link-drafts.json`
  accumulator and the **`damat module link-setup`** command no longer exist, and
  `module.json` no longer has a `link[]` field (see `@damatjs/module` ā€” its `link`
  manifest surface was removed, leaving the package unchanged from 0.3.6). A module
  now ships the `defineLink` file directly instead of declaring a rule. This flow was
  never published, so no installed app is affected.

## Action required

None ā€” drop-in upgrade. To ship a link from a module, add a `defineLink` file under
`links/models/` (import `defineLink` from `@damatjs/framework`). To activate a link a
module shipped: `damat module add <source>`, then
`damat-orm migrate:create link:<moduleId>` ā†’ `migrate:up` ā†’ `damat codegen <module>`.
To opt out, delete `src/links/<moduleId>/` before migrating.
